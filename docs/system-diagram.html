Grounded runtime foundation Local catalog database

The runtime app does not live-crawl the web. It uses the stored catalog built during ingestion.

  • Programs and program versions
  • Questions and question variants
  • Eligibility rules and amount rules
  • Sources, source snapshots, change events, review tasks
  • Screening sessions and session answers
Important boundary

Runtime screening and Explorer both operate on the grounded local catalog. Gemini helps during state ingestion and query interpretation, but it is not the final runtime eligibility engine.

How to read the diagram

The first half is data creation: official sources, crawl enrichment, Gemini-assisted state extraction, snapshots, and review tracking. The second half is runtime use: the local grounded catalog drives screening, Explorer, planning, and source-backed result cards.

Big architecture idea

The app is not a live web search engine during screening. It is a grounded rules-and-evidence product. It syncs source data ahead of time, stores programs, questions, rules, and sources locally, and then runs deterministic screening against that local catalog.

Current state caveat

The federal path is more deterministic because it starts from a structured official feed. The state path is grounded in official pages, but it is still less deterministic because it relies on Gemini to filter and extract structured state program data during ingestion.
